# HireFlow AI 招聘提效工作台

HireFlow 是一个面向 HR 招聘运营的 PC 端 AI 应用 Demo，用最小成本展示「企业微信 / 企业微信群 / 腾讯在线文档」招聘场景中的自动化落地能力：AI 自动抽取候选人信息、实时更新招聘台账、同步状态可视化、异常风险提醒，减少 HR 手工录入和反复追踪。

## 作业提交信息

- 代码仓库：[https://github.com/tangqb612-oss/HireFlow.git](https://github.com/tangqb612-oss/HireFlow.git)
- 本地 Demo：运行 `npm start` 后访问 `http://localhost:4173`
- 在线预览建议：启用 GitHub Pages，选择 `main` 分支根目录，预期地址为 `https://tangqb612-oss.github.io/HireFlow/`
- 技术栈：`HTML + CSS + JavaScript + Node.js`
- 成本策略：第一版使用无依赖 Node 后端模拟企业微信导入接口，先验证业务价值和交互闭环

## 背景与痛点

现有招聘流程中，HR 的日常工作分散在企业微信、企业微信群和腾讯在线文档中。候选人推荐、面试安排、负责人分配、Offer 进展等信息经常以聊天消息形式出现，之后再由 HR 手动整理到在线文档。

核心问题：

- 人工录入效率低：候选人、岗位、阶段、面试时间、负责人等字段需要重复复制和整理。
- 数据容易遗漏：群聊信息碎片化，重要进展容易被消息流淹没。
- 时效性差：腾讯在线文档依赖人工更新，招聘负责人无法实时掌握进展。
- 异常发现滞后：缺字段、重复建档、超时未跟进通常需要人工巡检。

## 方案目标

在最小成本预算下，建设一个可逐步落地的 AI 招聘数据助手，实现：

- 自动记录：从企业微信消息、群聊文本或手动粘贴内容中抽取候选人结构化信息。
- 实时同步：将招聘数据同步到腾讯在线文档式台账或后续真实腾讯文档。
- 数据可视化：自动生成招聘漏斗、候选人总量、待确认事项和同步状态。
- 智能提醒：识别缺失字段、重复候选人、面试未确认、超时未跟进等风险。
- 降低录入成本：让 HR 从手工维护表格转向只处理 AI 标记的异常和确认项。

## Demo 能力

当前 Demo 已实现一个 PC 端招聘提效工作台：

- 企业微信群聊消息导入模拟。
- AI 自动抽取候选人结构化字段。
- 招聘漏斗和核心指标自动刷新。
- 腾讯在线文档式台账实时更新。
- 缺失字段、重复候选人、待确认数据自动提醒。
- 支持按岗位和负责人筛选。
- 支持单独查看和编辑每位应聘者档案，包括姓名、性别、学历、毕业院校、院校层次/QS排名、电话、邮箱和当前岗位。
- 支持录入每个招聘阶段的表现评价、通过状态、原因，并标记是否停止该应聘者招聘流程。
- 支持人才储备库菜单，停止招聘流程后候选人从招聘台账移出，只在储备库保留档案、停止原因和最后阶段。
- 支持 AI 待办处理闭环，可查看档案、确认重复合并、确认缺失信息已补齐，并自动更新风险状态和同步状态。
- 支持周度/月度招聘情况查看，包括新增候选人、推进更新、Offer/入职、终止储备、同步情况、风险和负责人/渠道分布。

运行方式：

1. 克隆或下载本仓库。
2. 执行 `npm start`。
3. 浏览器访问 `http://localhost:4173`。
4. 点击「导入今日群聊」，前端会调用 `/api/wecom/import-today`，由后端返回模拟企业微信群聊和 AI 抽取结果。

如果直接用浏览器打开 `index.html`，页面会自动回退到本地模拟数据，但不会调用后端 API。

## 最小成本落地架构

### 1. 消息采集层

低成本优先级如下：

- Demo 阶段：前端内置企业微信群聊样例数据。
- 最小可实施版：Node 后端提供 `/api/wecom/import-today`，模拟企业微信今日群聊导入，前端点击按钮即可调用接口刷新页面。
- MVP 阶段：HR 复制群聊文本到页面，或通过企业微信群机器人收集固定格式消息。
- 增强阶段：在企业审批允许后接入企业微信会话存档、消息回调或 ATS 系统。

当前后端文件为 `server.js`，真实接入时替换其中 `/api/wecom/import-today` 的 mock 逻辑：

```text
1. 校验 HR 企业微信登录态
2. 根据 HR 绑定的招聘群读取今日群聊
3. 调用 AI 模型抽取候选人 JSON
4. 写入数据库/缓存
5. 返回 candidates 给前端
```

### 2. AI 结构化处理层

真实落地时，模型输入为企业微信消息、群聊文本或腾讯文档新增备注，输出统一 JSON：

```json
{
  "name": "王雨晴",
  "role": "产品经理",
  "school": "香港大学",
  "schoolTier": "海外院校QS排名26",
  "source": "内推",
  "stage": "已发offer",
  "schedule": "6 月 24 日入职",
  "owner": "Carol",
  "riskTags": ["Offer 跟进入职"],
  "confidence": 0.93
}
```

推荐处理策略：

- 高频字段先用规则抽取，如姓名、岗位、负责人、日期时间。
- 复杂语义再调用低成本大模型，例如阶段判断、风险标签、备注总结。
- 对低置信度、缺字段、疑似重复记录进入 HR 人工确认队列。
- 对手机号、微信号等敏感信息做脱敏展示和权限控制。

### 3. 数据同步层

- Demo 阶段：前端表格模拟腾讯在线文档台账。
- 最小可实施版：候选人修改通过 `PATCH /api/candidates/:id` 同步到 Node 后端内存状态。
- MVP 阶段：导出 CSV 或通过轻量服务写入腾讯在线文档/腾讯表格。
- 增强阶段：通过腾讯文档 API、云函数或企业内部服务实现自动双向同步。

### 4. 可视化与提醒层

- 招聘漏斗：简历待筛、通过初筛、待一面、待二面、已发offer、已入职、终止。
- 指标卡片：候选人总数、待 HR 确认、今日同步、风险提醒。
- 周/月报表：按候选人创建和更新时间统计招聘进展、阶段分布、负责人跟进量和渠道来源。
- 异常提醒：缺联系电话、面试时间未确认、重复候选人、超时未跟进。
- 筛选维度：岗位、负责人、招聘阶段、候选人来源。

## 成本控制

- Demo 阶段：零后端、零数据库、零模型调用成本。
- MVP 阶段：优先复用企业微信、腾讯在线文档和云函数。
- 模型调用只处理新增消息和变更内容，避免重复消耗 token。
- 高频字段用规则抽取，复杂语义再调用大模型。
- 人工只介入异常确认，不再维护全部招聘台账。

## 后续扩展
